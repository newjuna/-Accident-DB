v31_login_once_loading_cache

변경 내용
1. 웹앱 전체 로딩 구조 개선
   - 로그인 직후 Apps Script가 원본DB를 1회 읽고, 조회용 전체 데이터를 브라우저로 전달
   - 이후 대시보드 연도/월 변경, 데이터조회 월/영업부/팀/매장 검색, 차트 클릭, 상세보기는 브라우저 내부 데이터로 즉시 처리
   - 사용자가 신규 작성/저장하는 구조가 아니므로 로그인 후에는 서버 재조회 없이 화면 필터링을 우선 적용

2. 사고경위서 사고 영역
   - 대시보드 연도/월 변경 시 로딩창 없이 즉시 재계산
   - 데이터 조회 월 변경 시 서버 호출 제거
   - 영업부/팀/매장 검색 변경 시 서버 호출 제거
   - 차트/반복사고 매장 클릭 시 사고 리스트 서버 호출 제거
   - 상세보기는 이미 받은 데이터에서 즉시 표시

3. 산재 승인 사고 영역
   - 로그인 시 산재 승인 사고 전체 데이터를 함께 받아옴
   - 산재 승인 사고 대시보드/데이터조회/중상해 매장 메뉴 이동 시 서버 호출 제거
   - 연도/월/영업부/팀/매장 검색 변경 시 서버 호출 제거
   - 출퇴근재해 제외 기준 유지

4. 요약 생성은 기존 v29 방식 유지
   - 사고경위서 사고 요약 생성: summaryBatch API 사용
   - 산재 승인 사고 요약 생성: approvalSummary API 사용
   - 요약 생성과 이미지/ZIP 다운로드는 파일 생성 작업이므로 로딩창 유지

5. v28/v30 기존 기능 유지
   - Teams/카카오톡 썸네일 설정 유지
   - 사고 원본DB 안전보건팀 전용 노출 유지
   - 산재 승인 사고 엑셀 조건부 색상 유지

배포 방법
1. ZIP 압축 해제
2. GitHub Pages 저장소에 index.html / app.js / style.css / logo.png / thumbnail 관련 파일 전체 덮어쓰기
3. Apps Script의 Code.gs도 반드시 v31 Code.gs로 전체 교체
4. Apps Script 저장 후 새 배포 또는 기존 웹앱 배포 업데이트
5. GitHub Pages 주소에서 강력 새로고침 후 테스트

중요
- app.js만 바꾸면 안 됩니다. Code.gs의 startup 응답에 rows / approvalRows가 추가되어야 v31 효과가 납니다.
- 로그인 후 원본DB가 바뀌어도 현재 화면에는 즉시 반영되지 않습니다. 최신 데이터를 보려면 새로고침 또는 재로그인을 해야 합니다.
- 대신 월 변경/필터 변경/메뉴 이동 로딩은 크게 줄어듭니다.

테스트 순서
1. 안전보건팀 로그인
2. 로그인 초기 로딩 후 대시보드 연도/월을 여러 번 변경
3. 사고경위서 사고 > 데이터 조회에서 월/영업부/팀/매장 검색 변경
4. 차트 또는 반복사고 매장 클릭 후 상세보기 열기
5. 산재 승인 사고 > 대시보드/데이터조회/중상해 매장 메뉴 이동
6. 산재 승인 사고 연도/월 변경
7. 요약 생성 시에만 로딩창이 뜨는지 확인
